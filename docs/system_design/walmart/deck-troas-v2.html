TROAS Deep Dive

Target Return on Ad Spend — automatic bid adjustment to hit an advertiser-configured revenue target

TROAS

The Problem TROAS Solves

Without TROAS — Manual Bidding

  • Advertiser sets a static CPC bid per keyword
  • 10M+ ads × thousands of keywords = impossible to tune
  • Static bids ignore time-of-day, competition, seasonality
  • Either overspends (poor ROAS) or misses impressions
  • Requires constant manual monitoring and adjustment

The Scale Problem

A human cannot compute the right bid for 10M campaigns at 12,000 auctions/second. Even rule-based systems can't adapt to real-time demand signals.

With TROAS — Auto Bidding

  • Advertiser sets one number: targetRoas = 4.0
  • System computes the right bid per auction in real-time
  • Maximizes spend while respecting the ROAS constraint
  • Adapts using offline ML models + live budget signals
  • Learns campaign performance over time (phases)

TROAS Constraint

targetRoas = 4.0 means: for every $1 spent on ads, generate $4 in product revenue. The system maximises impressions subject to this constraint.

Bidding Engine Cache — Key Design Decisions

Why 3 Separate Caches?

  • Different refresh rates: ad-group params change less often than item-level bid params
  • Day-part factors rotate every 24h on a fixed schedule
  • Allows per-source-type ConfigSvc version validation

Zero-Key Fallback Hierarchy

For item bid params:

1. (adGroupId, itemId) — item override
2. (adGroupId, '0') — group default
3. ('0', '0') — system default
4. ConfigSvc defaults — hard-coded fallback

ConfigSvc Version Validation

  • Every cache entry carries a version field
  • ConfigSvc controls "active" version per source type
  • Mismatch → VERSION_NOT_IN_CACHE + ConfigSvc default used
  • Zero-downtime rollout: stage DB → flip ConfigSvc key → instant activation
  • Per-source-type granularity: only affected params switch

Zero-Key Fallback — Scale Decision

  • Most items in an ad group share identical bid params
  • Write one group default at (adGroupId, '0') — not per item
  • Only items needing a per-item override get their own row
  • item_bid_params stays at O(campaigns + overrides), not O(campaigns × items)

ML Pipeline — Offline Training + Online Inference

TROAS · ML Platform

ML Scoring Architecture — Key Decisions

Two-Tower Retrieval

  • Ad Tower: offline at ad creation → 128-dim embedding stored in VectorDB/Vespa
  • User Tower: online at serving → Triton inference → ANN top-100 candidates
  • Decouples index build from serving; scales to 10M ads without per-request indexing

DCN v2 Ranking

  • Ranks top-100 candidates from retrieval
  • Cross network (feature interactions) + Deep network (non-linear)
  • Outputs pCTR, pCVR scores used in bid computation

4-Level Inference Cache

  • L1: Caffeine (in-process) — sub-microsecond
  • L2: ClusterCache (cluster) — ~1ms
  • L3: Cassandra (persistent) — ~5ms
  • L4: Triton H100 (live inference) — ~10ms
  • Cache hit rate on L1/L2 determines if H100s are needed

Feast Feature Store

  • 5 FeatureViews: item_features, cnsldv2, items_rate, pvpc, item_quality
  • Offline: PySpark on Dataproc → BigQuery → Cassandra (daily)
  • Online: Feast SDK → Cassandra ads keyspace → <10ms

Budget Pacing — Key Design Decisions

BudgetSvc Architecture

  • Azure SQL: durable daily budget state — authoritative spend ledger
  • Daily rollover at midnight per tenant timezone (WMT, SAMS, INTL separate cycles)
  • Budget status written to Cassandra → IndexPublisher → Solr index updated
  • Over-budget: campaign removed from index. In-budget: eligible for auction

Budget Splitter — A/B Attribution

  • Campaign index carries budgetBucketId per candidate
  • AdOrchestrator resolves bucket from ExperimentPlatform experiment config
  • Click beacon: bkt = "sp_wmt_srp|bb_BUCKET_A"
  • Enables per-bucket spend attribution without extra lookups
  • TROAS_SCORING_CALL{bbid, plmtck, modelId} — dimensional metric

Budget Eligibility Propagation

Over-budget campaigns are removed from the Solr/Vespa index via: BudgetSvc writes to Cassandra → IndexPublisher reads → index updated. Eventually consistent with ~minutes lag.

TROAS · Resilience

Safety Guards & Edge Cases

Operational Guards

NO_BP bidPrime missing → ConfigSvc default → skip item if still absent
ZERO_TR targetRoas=0 guard before ML formula
NO_ML_PARAMS ML params unavailable → ROAS-only fallback
NO_EXPLORE_CONFIG Exploration distributions missing → ConfigSvc default distributions
NO_PF pacingFactor absent → default 1.0 (no pacing reduction)
INVALID_WEIGHTS Bid weights absent/invalid → ROAS-only path
UNKNOWN_PHASE Unrecognised phase → INIT (safe conservative fallback)
BELOW_FLOOR Bid below $0.20 floor → clamp + metric emitted

Async / Timeout Guards

TIMEOUT processTroasRequest async timeout → item excluded from response
VERSION_MISMATCH All source types stale → all ConfigSvc defaults; bid still proceeds
CACHE_MISS New campaign: cache not yet seeded → zero-key fallback → ConfigSvc

Protobuf Contract Guards

DESER_FAIL bidParamsNode JSON parse fail → partially populated params → individual ConfigSvc fallbacks
NO_TROAS_NODE AuctionEngine receives no TROAS params → standard CPC bidding for that candidate
Key principle: Never let a missing parameter crash an auction. Every parameter has a fallback, every fallback has a metric, every metric has an alert.

Open Problems

  • Cold-start convergence — new campaigns wait for bidPrime to be available; exploration phase can overbid
  • OpenTelemetry traces don't span Kafka async boundaries — end-to-end latency invisible

What I'd Change

  • Complete Solr → Vespa migration: CandidateSvc still spans both; Vespa native ANN is faster
  • Bidding Engine source of truth → Cassandra for horizontal scale (MySQL bottleneck at extreme load)
  • Multi-signal exploration: incorporate post-click engagement signals for faster phase convergence
Summary

Summary — TROAS in One Flow

// Ingestion (async)
Advertiser → CampaignSvc → Kafka → IndexPublisher → Solr/Vespa

// Serving (sync, <50ms)
Shopper → AdGateway → AdOrchestrator → BiddingEngine
BiddingEngine loads 3 Caffeine caches // phase + params
BiddingEngine → AuctionEngine (bidParamsNode JSON)

// Bid per phase (AuctionEngine)
INIT/TRANS: base bid × time × pacing
LEARNING: explore bid space (Bayesian)
PAUSED: no bid emitted
OPT: ML bid ← Triton pCTR/pCVR

finalCPC = clamp(bid, $0.20, $5.00) → round(4dp)
TSP auction → Top-N ads → Shopper

Key Properties

  • Advertiser sets one number: targetRoas
  • 5-phase lifecycle: exploration → ML-driven bidding
  • 3-cache bidding engine: in-process, zero-serialization
  • 4-level ML cache: H100s only for cache misses
  • Every parameter has a ConfigSvc fallback + metric
